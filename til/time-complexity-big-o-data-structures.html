시간복잡도, 빅오 표기법, 대표 자료구조 정리

TIL · 알고리즘과 자료구조 · 2026.07.09

알고리즘을 비교할 때는 실행 시간이 몇 초인지보다 입력이 커질수록 연산량이 어떻게 증가하는지를 보는 것이 중요합니다. 시간복잡도와 빅오 표기법은 그 증가 추세를 간단히 표현하고, 자료구조 선택은 그 비용을 실제 코드에서 결정합니다.

시간복잡도란

시간복잡도는 입력 크기 n이 커질 때 알고리즘의 연산 횟수가 어떤 비율로 증가하는지를 나타내는 개념입니다. 같은 기능을 하는 코드라도 입력이 10개일 때는 차이가 작아 보일 수 있지만, 입력이 10만 개나 100만 개가 되면 반복문 구조와 자료구조 선택에 따라 실행 시간이 크게 달라집니다.

시간복잡도는 실제 초 단위 시간을 그대로 말하지 않습니다. 컴퓨터 성능, 언어, 컴파일러, 메모리 상태에 따라 실제 시간은 달라질 수 있기 때문입니다. 대신 입력이 증가할 때 연산량이 상수처럼 유지되는지, 선형으로 증가하는지, 제곱으로 증가하는지 같은 성장 형태를 봅니다.

핵심 질문은 "입력 크기가 두 배가 되면 이 코드는 얼마나 더 많은 일을 하는가?"입니다. 이 질문에 답하면 작은 예제에서는 보이지 않던 성능 차이를 미리 예측할 수 있습니다.

빅오 표기법

빅오 표기법은 알고리즘의 성장률을 가장 큰 영향이 있는 항만 남겨 표현하는 방식입니다. 예를 들어 어떤 코드의 연산량이 3n + 10이라면 입력이 커질수록 상수 10과 계수 3보다 n이 증가한다는 사실이 더 중요합니다. 그래서 보통 O(n)으로 표현합니다.

n² + n + 1처럼 여러 항이 있을 때도 입력이 커질수록 가장 빠르게 증가하는 항이 전체 성능을 지배합니다. 이 경우에는 O(n²)로 봅니다. 빅오는 정확한 실행 시간을 계산하기 위한 도구라기보다, 알고리즘이 커지는 입력을 감당할 수 있는지 판단하는 기준에 가깝습니다.

대표 복잡도

  • O(1) · 상수 시간

    입력 크기와 상관없이 일정한 횟수로 끝납니다. 배열의 인덱스 접근, 스택의 top 확인이 대표적입니다.

  • O(log n) · 로그 시간

    한 번 처리할 때마다 탐색 범위가 크게 줄어듭니다. 정렬된 배열에서의 이진 탐색이 대표적입니다.

  • O(n) · 선형 시간

    입력 크기에 비례해 연산량이 증가합니다. 배열 전체 순회, 선형 탐색이 여기에 해당합니다.

  • O(n log n)

    효율적인 정렬 알고리즘에서 자주 등장합니다. 병합 정렬, 힙 정렬, 평균적인 퀵 정렬을 떠올릴 수 있습니다.

  • O(n²) · 제곱 시간

    중첩 반복문처럼 모든 원소 쌍을 비교할 때 자주 나옵니다. 입력이 커지면 비용이 빠르게 증가합니다.

  • O(2ⁿ), O(n!)

    모든 경우의 수를 탐색하는 완전 탐색에서 등장할 수 있습니다. 작은 입력에서는 가능하지만 큰 입력에서는 금방 한계에 부딪힙니다.

대표 자료구조

자료구조는 데이터를 어떤 형태로 저장하고 접근할지 정하는 방식입니다. 같은 데이터라도 어떤 자료구조를 쓰느냐에 따라 탐색, 삽입, 삭제, 정렬, 우선순위 처리의 비용이 달라집니다.

배열 · Array

배열은 연속된 메모리 공간에 같은 종류의 데이터를 순서대로 저장합니다. 인덱스를 알고 있으면 O(1)에 접근할 수 있어 빠르지만, 중간에 삽입하거나 삭제하면 뒤의 원소들을 이동해야 해서 보통 O(n)이 듭니다. 순서가 중요하고 인덱스로 자주 접근하는 데이터에 잘 맞습니다.

연결 리스트 · Linked List

연결 리스트는 각 노드가 데이터와 다음 노드의 위치를 함께 들고 있는 구조입니다. 노드 위치를 이미 알고 있다면 삽입과 삭제는 빠르게 처리할 수 있지만, 특정 인덱스로 바로 접근할 수 없어서 탐색은 보통 O(n)입니다. 잦은 삽입과 삭제가 필요하고 순차 접근이 중심일 때 고려할 수 있습니다.

스택 · Stack

스택은 나중에 들어온 데이터가 먼저 나가는 LIFO 구조입니다. push, pop, top은 보통 O(1)에 처리됩니다. 함수 호출 관리, 되돌리기, 괄호 검사, 깊이 우선 탐색 같은 상황에서 자주 사용됩니다.

큐 · Queue

큐는 먼저 들어온 데이터가 먼저 나가는 FIFO 구조입니다. 삽입과 제거는 보통 O(1)에 처리되며, 작업 대기열, 이벤트 처리, 너비 우선 탐색에 잘 어울립니다. 우선순위가 필요한 경우에는 일반 큐 대신 우선순위 큐를 사용합니다.

해시 테이블 · Hash Table

해시 테이블은 키를 해시 함수로 변환해 데이터를 저장하는 구조입니다. 평균적으로 탐색, 삽입, 삭제가 O(1)에 가깝지만, 해시 충돌이 많거나 해시 함수가 좋지 않으면 성능이 나빠질 수 있습니다. 빠른 검색이 필요한 딕셔너리, 캐시, 중복 검사에 자주 사용됩니다.

트리 · Tree

트리는 계층 관계를 표현하는 자료구조입니다. 이진 탐색 트리는 정렬된 규칙을 이용해 탐색을 줄일 수 있지만, 한쪽으로 치우치면 O(n)까지 나빠질 수 있습니다. 균형 잡힌 트리는 탐색, 삽입, 삭제를 보통 O(log n)에 처리할 수 있어 정렬된 데이터 관리에 유리합니다.

그래프 · Graph

그래프는 정점과 간선으로 관계를 표현하는 구조입니다. 길 찾기, 네트워크, 스킬 트리, 의존성 관계처럼 여러 대상이 복잡하게 연결된 문제에 사용됩니다. 인접 리스트는 간선이 적은 그래프에 효율적이고, 인접 행렬은 두 정점의 연결 여부를 빠르게 확인하기 좋습니다.

힙 · Heap

힙은 최댓값이나 최솟값을 빠르게 꺼내기 위한 완전 이진 트리 기반 구조입니다. 최상위 원소 확인은 O(1), 삽입과 삭제는 보통 O(log n)에 처리됩니다. 우선순위 큐, 작업 스케줄링, 힙 정렬에서 자주 쓰입니다.

자료구조 선택 기준

자료구조를 고를 때는 "가장 유명한 구조"보다 "내 코드에서 가장 자주 일어나는 연산"을 먼저 봐야 합니다. 데이터를 인덱스로 자주 읽는다면 배열이 단순하고 빠릅니다. 키로 빠르게 찾는 일이 많다면 해시 테이블이 좋습니다. 항상 가장 작은 값이나 큰 값을 꺼내야 한다면 힙이 자연스럽습니다.

게임 개발에서도 이 기준은 그대로 적용됩니다. 매 프레임 순회해야 하는 오브젝트 목록은 연속 저장 구조가 유리할 수 있고, 아이템 ID로 정보를 자주 찾는다면 해시 기반 맵이 편합니다. 길 찾기나 상태 전이처럼 관계가 중요한 문제는 그래프로 모델링하면 문제를 더 명확하게 다룰 수 있습니다.

정리

  • 시간복잡도는 입력 크기가 커질 때 연산량이 어떻게 증가하는지 보는 기준이다.
  • 빅오 표기법은 가장 큰 성장 항만 남겨 알고리즘의 확장성을 표현한다.
  • 배열은 인덱스 접근이 빠르고, 연결 리스트는 위치를 알고 있을 때 삽입과 삭제가 유리하다.
  • 스택과 큐는 데이터가 들어오고 나가는 순서를 제한해 문제를 단순하게 만든다.
  • 해시 테이블은 평균적으로 빠른 검색을 제공하고, 트리와 힙은 정렬과 우선순위 처리에 강하다.
  • 그래프는 복잡한 관계와 경로 문제를 표현하는 데 적합하다.

결국 시간복잡도는 알고리즘의 성장 방식을 읽는 언어이고, 자료구조는 그 성능을 현실의 코드에서 만들어내는 선택입니다. 둘을 함께 보면 "동작하는 코드"를 넘어 "입력이 커져도 버틸 수 있는 코드"를 설계할 수 있습니다.
