SOLID 원칙으로 설계 책임 나누기

TIL · 객체지향 설계 · 2026.07.08

SOLID는 객체지향 설계를 더 오래 유지하고 쉽게 확장하기 위한 다섯 가지 기준입니다. 단순히 원칙 이름을 외우는 것이 아니라, 변경이 생겼을 때 어떤 코드가 흔들리는지 판단하는 도구로 쓰는 것이 중요합니다.

SOLID가 필요한 이유

객체지향 프로그래밍은 추상화, 캡슐화, 상속, 다형성을 통해 복잡한 문제를 객체들의 협력으로 나눠 다루는 방식입니다. 하지만 클래스를 많이 만든다고 해서 자동으로 좋은 구조가 되는 것은 아닙니다. 한 클래스가 너무 많은 일을 하거나, 구체적인 구현에 강하게 묶이거나, 상속 관계가 기대와 다르게 동작하면 오히려 변경이 어려운 코드가 됩니다.

SOLID는 이런 문제를 줄이기 위한 설계 원칙입니다. 각 원칙은 서로 다른 관점에서 "변경에 강한 구조"를 만들도록 돕습니다. 게임 개발로 비유하면 캐릭터, 스킬, 아이템, UI, 저장 시스템처럼 규칙이 계속 늘어나는 영역에서 특히 유용합니다.

  • SRP · Single Responsibility Principle

    클래스는 하나의 책임, 하나의 변경 이유를 가져야 합니다.

  • OCP · Open Closed Principle

    기능 확장에는 열려 있고, 기존 코드 수정에는 닫혀 있어야 합니다.

  • LSP · Liskov Substitution Principle

    하위 타입은 상위 타입을 대체해도 프로그램의 기대 동작을 깨뜨리지 않아야 합니다.

  • ISP · Interface Segregation Principle

    클라이언트가 사용하지 않는 기능에 의존하지 않도록 인터페이스를 작게 분리해야 합니다.

  • DIP · Dependency Inversion Principle

    구체적인 구현보다 추상화에 의존해 변경의 영향을 줄여야 합니다.

SRP · 단일 책임 원칙

단일 책임 원칙은 클래스가 단 하나의 책임에 집중해야 한다는 원칙입니다. 여기서 책임은 단순히 "메서드가 하나"라는 뜻이 아니라, 코드가 변경되는 이유를 말합니다. 같은 이유로 함께 바뀌는 기능은 한곳에 모을 수 있지만, 서로 다른 이유로 바뀌는 기능은 분리하는 편이 좋습니다.

예를 들어 플레이어 클래스가 이동, 전투 계산, 인벤토리 관리, UI 출력, 저장 파일 기록까지 모두 담당한다면 작은 변경도 플레이어 클래스 전체를 건드리게 됩니다. 이동 규칙이 바뀌어도, 저장 포맷이 바뀌어도, UI 표시 방식이 바뀌어도 같은 클래스를 수정해야 하므로 코드가 쉽게 불안정해집니다.

SRP를 적용할 때의 핵심 질문은 "이 클래스가 바뀌는 이유는 몇 개인가?"입니다. 답이 여러 개라면 이동 처리, 전투 처리, 저장 처리처럼 책임을 나눌 후보가 있다는 뜻입니다.

OCP · 개방 폐쇄 원칙

개방 폐쇄 원칙은 새로운 기능을 추가할 수 있어야 하지만, 이미 검증된 기존 코드는 가능한 적게 수정해야 한다는 원칙입니다. "확장에는 열려 있고, 수정에는 닫혀 있다"는 말은 기존 코드를 절대 수정하지 말라는 뜻이 아니라, 새로운 요구사항이 들어올 때마다 핵심 로직을 계속 뜯어고치는 구조를 피하자는 의미입니다.

예를 들어 스킬 타입이 추가될 때마다 전투 계산 함수 안에 `if`나 `switch`가 계속 늘어난다면, 새 스킬을 추가할 때마다 기존 전투 로직을 건드리게 됩니다. 반대로 공통 스킬 인터페이스를 두고 각 스킬이 자신의 효과 계산을 맡도록 만들면, 새 스킬 클래스를 추가하는 방식으로 확장할 수 있습니다.

OCP는 추상화와 다형성을 잘 사용할 때 힘을 발휘합니다. 공통 규칙은 안정적인 인터페이스로 묶고, 달라지는 부분은 새로운 구현으로 추가할 수 있게 만드는 것이 핵심입니다.

LSP · 리스코프 치환 원칙

리스코프 치환 원칙은 하위 타입 객체가 상위 타입 객체를 대신해도 프로그램이 기대한 방식으로 동작해야 한다는 원칙입니다. 상속을 사용하면 부모 타입 변수에 자식 객체를 넣을 수 있지만, 타입만 맞는다고 항상 좋은 상속 구조가 되는 것은 아닙니다.

예를 들어 `Character` 타입은 `Attack()`을 호출하면 공격이 수행된다고 기대한다고 가정합니다. 그런데 어떤 하위 클래스가 `Attack()` 호출 시 예외를 던지거나, 아무 동작도 하지 않거나, 전혀 다른 자원 소비 규칙을 강제로 요구한다면 상위 타입을 사용하는 코드는 더 이상 안전하게 동작하지 않습니다.

LSP를 지키려면 오버라이딩할 때 부모가 약속한 의미를 깨뜨리지 않아야 합니다. 자식 클래스가 부모보다 더 강한 사전 조건을 요구하거나, 부모가 보장하던 결과를 보장하지 못하면 치환 가능성이 깨집니다. 상속이 어색하다면 상속보다 합성이나 인터페이스 분리를 고려하는 편이 낫습니다.

ISP · 인터페이스 분리 원칙

인터페이스 분리 원칙은 클라이언트가 자신이 사용하지 않는 기능에 의존하지 않아야 한다는 원칙입니다. SRP가 클래스의 책임을 나누는 기준이라면, ISP는 인터페이스의 책임을 나누는 기준에 가깝습니다.

예를 들어 모든 게임 오브젝트가 `Move()`, `Attack()`, `Talk()`, `OpenInventory()`, `Save()`를 가진 거대한 인터페이스를 구현해야 한다면 문제가 생깁니다. 문, NPC, 투사체, 플레이어는 필요한 기능이 서로 다릅니다. 사용하지 않는 메서드를 억지로 구현하게 되면 빈 함수, 의미 없는 예외, 불필요한 의존성이 늘어납니다.

더 나은 방식은 `IMovable`, `IAttackable`, `IInteractable`, `ISavable`처럼 사용 목적에 맞게 인터페이스를 나누는 것입니다. 그러면 각 객체는 자신에게 필요한 계약만 구현하고, 해당 기능을 사용하는 코드는 필요한 인터페이스에만 의존할 수 있습니다.

DIP · 의존 역전 원칙

의존 역전 원칙은 상위 수준의 정책 코드가 하위 수준의 구체 구현에 직접 의존하지 않도록 하자는 원칙입니다. 쉽게 말하면 "구현 클래스에 바로 묶이지 말고, 인터페이스나 추상 클래스 같은 안정적인 약속에 의존하라"는 뜻입니다.

예를 들어 전투 시스템이 특정 `FireballSkill` 클래스에 직접 의존하면, 얼음 스킬이나 회복 스킬을 추가할 때 전투 시스템을 계속 수정해야 할 수 있습니다. 반대로 전투 시스템이 `ISkill` 같은 추상화에 의존하고, 실제 스킬 구현이 그 인터페이스를 따르게 만들면 전투 시스템은 구체 스킬의 종류를 몰라도 됩니다.

DIP는 테스트와 유지보수에도 도움이 됩니다. 저장 시스템, 사운드 시스템, 네트워크 통신처럼 외부 환경에 가까운 코드를 추상화 뒤에 두면, 핵심 로직은 더 안정적으로 유지되고 테스트용 구현으로 바꿔 끼우기도 쉬워집니다.

정리

SOLID는 모든 코드에 기계적으로 적용하는 규칙이라기보다, 설계가 흔들릴 때 확인할 수 있는 체크리스트에 가깝습니다. 작은 프로젝트에서 처음부터 모든 구조를 복잡하게 나누면 오히려 부담이 될 수 있습니다. 대신 변경이 반복되는 지점, 조건문이 계속 늘어나는 지점, 테스트하기 어려운 지점부터 원칙을 적용해보는 것이 현실적입니다.

  • SRP는 책임과 변경 이유를 분리한다.
  • OCP는 새 기능을 추가할 때 기존 핵심 코드를 덜 건드리게 한다.
  • LSP는 상속 관계가 실제로 안전하게 대체 가능한지 확인한다.
  • ISP는 사용하지 않는 기능에 억지로 의존하지 않게 한다.
  • DIP는 구체 구현보다 안정적인 추상화에 의존하게 한다.

결국 SOLID의 목적은 코드를 예쁘게 나누는 것이 아니라, 다음 변경을 더 작고 안전하게 만드는 것입니다. 특히 게임 로직처럼 캐릭터, 스킬, 아이템, 상태 효과가 계속 추가되는 코드에서는 책임과 의존 방향을 일찍 점검해두는 것만으로도 유지보수 비용을 크게 줄일 수 있습니다.
