분기 예측으로 이해하는 CPU 파이프라인 최적화

TIL · 컴퓨터 구조 · 2026.07.09

분기 예측은 CPU가 조건문의 결과를 기다리지 않고 다음에 실행될 가능성이 높은 명령어를 미리 처리하는 기술입니다. 예측이 맞으면 파이프라인이 계속 흐르지만, 예측이 틀리면 준비하던 작업을 버리고 다시 시작해야 하므로 성능 손실이 커집니다.

CPU 파이프라인과 분기

CPU는 명령어 하나를 끝까지 처리한 뒤 다음 명령어를 시작하는 방식으로만 동작하지 않습니다. 명령어 가져오기, 해석하기, 실행하기, 메모리 접근하기, 결과 쓰기 같은 단계를 파이프라인으로 나누고, 여러 명령어를 서로 다른 단계에 동시에 올려 처리합니다. 공장 라인처럼 각 단계가 겹쳐 돌아가기 때문에 처리량을 높일 수 있습니다.

문제는 if, switch, 반복문처럼 실행 흐름이 갈라지는 분기문을 만났을 때 생깁니다. 분기 조건의 결과가 확정되기 전까지는 다음에 어느 명령어를 가져와야 하는지 알 수 없습니다. 이때 CPU가 매번 멈춰서 기다리면 파이프라인이 비고, 빠르게 동작하도록 만든 구조의 장점이 줄어듭니다. 

if (x > 0) {
  a();
} else {
  b();
}

위 코드에서 CPU는 x > 0의 결과가 나오기 전까지 a() 쪽 명령어를 준비해야 할지, b() 쪽 명령어를 준비해야 할지 확신할 수 없습니다. 분기 예측은 이 빈 시간을 줄이기 위해 등장한 기술입니다.

분기 예측이 하는 일

분기 예측은 CPU가 "이번 조건문은 참일 가능성이 높다", "이 반복문은 아직 계속 돌 가능성이 높다"처럼 다음 실행 경로를 미리 추측하는 과정입니다. 예측한 경로의 명령어를 먼저 가져오고 실행 준비를 해두면, 실제 결과가 예측과 맞았을 때 기다리는 시간이 줄어듭니다.

중요한 점은 분기 예측이 프로그램의 결과를 바꾸는 기능이 아니라는 것입니다. CPU는 예측 경로를 먼저 처리하더라도, 최종적으로 조건 결과가 맞는지 확인합니다. 예측이 맞으면 준비한 작업을 그대로 이어가고, 틀리면 잘못된 경로에서 진행한 작업을 버린 뒤 올바른 경로로 다시 실행합니다.

분기문 자체가 항상 느린 것은 아닙니다. 성능 손실의 핵심은 예측하기 어려운 분기가 반복되어 CPU가 자주 잘못된 경로를 준비하게 되는 상황입니다.

예측 실패와 파이프라인 플러시

분기 예측이 틀리면 CPU는 이미 파이프라인에 올려둔 잘못된 경로의 명령어들을 폐기해야 합니다. 이 과정을 파이프라인 플러시라고 부릅니다. 잘못 준비한 작업을 버리고, 실제로 실행해야 하는 주소에서 명령어를 다시 가져와야 하므로 그동안 파이프라인의 일부 단계가 비게 됩니다.

예측 실패 비용은 CPU 구조와 상황에 따라 달라지지만, 깊은 파이프라인일수록 버려야 하는 준비 작업이 많아질 수 있습니다. 그래서 최신 CPU는 분기 예측 정확도를 높이기 위해 실행 기록, 분기 주소, 패턴 정보 등을 활용합니다. 예측이 잘 맞는 반복문과 조건문은 빠르게 흐르지만, 결과가 무작위에 가까운 조건은 CPU가 맞히기 어렵습니다.

  • 예측 성공

    예상한 경로가 실제 경로와 같아 파이프라인을 거의 그대로 이어갈 수 있습니다.

  • 예측 실패

    잘못된 경로의 명령어를 버리고 올바른 경로로 다시 채워야 해서 지연이 발생합니다.

정적 예측과 동적 예측

분기 예측은 크게 정적 분기 예측과 동적 분기 예측으로 나눠 볼 수 있습니다. 둘의 차이는 실행 중에 얻은 기록을 활용하는지 여부입니다.

  • 정적 분기 예측

    실행 기록을 보지 않고 고정된 규칙으로 예측합니다. 예를 들어 특정 방향의 분기는 taken으로 본다거나, 반복문처럼 뒤로 점프하는 분기는 계속 반복될 가능성이 높다고 보는 식입니다. 구조가 단순하지만 실제 데이터 패턴을 반영하지 못한다는 한계가 있습니다.

  • 동적 분기 예측

    CPU가 과거 실행 기록을 보고 다음 결과를 예측합니다. 같은 조건문이 최근에 참이었는지 거짓이었는지, 특정 패턴이 반복되는지 같은 정보를 활용합니다. 현대 CPU는 복잡한 동적 예측기를 사용해 자주 반복되는 분기의 흐름을 높은 확률로 맞히려고 합니다.

코드에서 생각할 점

일반적인 애플리케이션 코드에서 모든 조건문을 분기 예측 관점으로 직접 최적화할 필요는 없습니다. 먼저 읽기 쉬운 코드와 올바른 알고리즘을 작성하는 것이 우선입니다. 다만 같은 코드가 매우 많이 반복되는 루프, 대량의 데이터를 처리하는 정렬·필터링, 게임 루프처럼 프레임마다 실행되는 경로에서는 분기 예측 실패가 성능에 영향을 줄 수 있습니다.

예측하기 쉬운 분기는 보통 일정한 패턴을 가집니다. 대부분 참인 조건, 대부분 거짓인 조건, 반복 횟수가 안정적인 루프는 CPU가 학습하기 쉽습니다. 반대로 데이터가 무작위라 조건 결과가 계속 바뀌면 예측 정확도가 낮아질 수 있습니다. 이 경우에는 자료 배치, 조건 순서, 알고리즘 변경, 불필요한 분기 제거가 도움이 될 수 있습니다.

성능 최적화는 측정 이후에 하는 편이 안전합니다. 분기 예측을 의식하되, 실제 병목인지 확인하지 않고 코드를 복잡하게 바꾸면 유지보수 비용만 커질 수 있습니다.

정리

  • CPU는 파이프라인을 통해 여러 명령어를 겹쳐 실행하며 처리량을 높인다.
  • 분기문은 다음 실행 경로를 즉시 알 수 없게 만들어 파이프라인 흐름을 방해할 수 있다.
  • 분기 예측은 조건 결과를 미리 추측해 예상 경로의 명령어를 먼저 준비하는 기술이다.
  • 예측이 맞으면 빠르게 이어가지만, 틀리면 파이프라인 플러시로 잘못 준비한 작업을 버려야 한다.
  • 정적 예측은 고정 규칙을 사용하고, 동적 예측은 과거 실행 기록과 패턴을 활용한다.
  • 코드 최적화에서는 먼저 측정하고, 반복 실행되는 핵심 경로에서 예측하기 어려운 분기를 줄이는 것이 중요하다.

분기 예측을 이해하면 조건문이 단순한 문법을 넘어 CPU 실행 흐름과 연결되어 있다는 점을 볼 수 있습니다. 특히 성능이 중요한 루프와 실시간 게임 로직에서는 "어떤 코드가 실행되는가"뿐 아니라 "CPU가 그 흐름을 얼마나 잘 예상할 수 있는가"도 생각해볼 만합니다.
