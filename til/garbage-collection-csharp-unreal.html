C#과 언리얼로 이해하는 가비지 컬렉션

TIL · 메모리 관리 · 2026.07.14

가비지 컬렉터는 더 이상 도달할 수 없는 관리 객체의 메모리를 자동으로 회수합니다. 하지만 C#의 관리 힙과 언리얼의 UObject는 추적 방식과 사용 규칙이 다르며, 파일·소켓 같은 자원이나 일반 C++ 객체까지 대신 정리해 주지는 않습니다.

가비지 컬렉션이 필요한 이유

가비지 컬렉션(Garbage Collection, GC)은 프로그램이 더 이상 사용할 수 없는 관리 객체를 찾아 그 메모리를 회수하는 런타임 기능입니다. 수동 메모리 관리에서는 할당과 해제의 짝을 개발자가 직접 맞춰야 합니다. 

Enemy* enemy = new Enemy();

delete enemy;
enemy = nullptr;

delete를 빠뜨리면 사용하지 않는 메모리가 계속 남는 메모리 누수가 생깁니다. 반대로 삭제된 주소를 다시 사용하면 댕글링 포인터를 역참조하게 되어 정의되지 않은 동작이 발생합니다. 

delete enemy;
enemy->Attack(); // 삭제된 객체에 접근: 정의되지 않은 동작

GC가 있는 환경에서는 런타임이나 엔진이 관리 대상 객체의 메모리 회수를 담당해 이런 부담을 줄입니다. 다만 사용하지 않는 객체라도 프로그램 어딘가에서 계속 참조하고 있다면 GC는 살아 있는 객체로 판단합니다. GC는 메모리 누수를 불가능하게 만드는 기능이 아니라, 수동 해제를 도달 가능성 추적으로 바꾸는 기능에 가깝습니다.

도달 가능성으로 생존 판단하기

추적형 GC는 단순한 참조 횟수보다 GC Root에서 객체에 도달할 수 있는가를 기준으로 객체의 생존을 판단합니다. 실행 중인 스레드의 스택과 지역 변수, 정적 필드처럼 런타임이 시작점으로 삼는 참조가 GC Root가 됩니다. 살아 있는 객체가 다른 객체를 참조한다면 그 참조를 따라 연결된 객체도 도달 가능한 것으로 표시합니다.

GC Root → 객체 A → 객체 B처럼 참조 경로가 하나라도 남아 있으면 B도 수집되지 않습니다. 반대로 객체끼리 연결되어 있어도 Root에서 시작하는 경로가 없으면 수집 대상이 될 수 있습니다.

class A { public B Other; }
class B { public A Other; }

A a = new A();
B b = new B();

a.Other = b;
b.Other = a;

a = null;
b = null;

마지막 두 대입 이후 다른 참조가 없다면 A와 B는 서로를 가리키더라도 GC Root에서 도달할 수 없습니다. 따라서 도달 가능성을 추적하는 C# GC는 이런 순환 참조도 정상적으로 수집할 수 있습니다.

C#의 관리 힙과 세대별 GC

C#에서 일반적인 참조 형식 객체는 관리 힙에 놓이고, 변수에는 객체 자체가 아니라 그 객체를 가리키는 참조가 저장됩니다.

class Enemy
{
    public string Name;
}

Enemy enemy = new Enemy();
enemy.Name = "Goblin";
enemy = null;

enemy = null은 참조 하나를 끊을 뿐 객체를 즉시 삭제하라는 명령이 아닙니다. 다른 도달 경로가 없다면 객체는 수집 가능한 상태가 되고, 실제 회수 시점은 런타임이 결정합니다. 또한 JIT 최적화에 따라 지역 변수의 유효 범위와 실제 도달 가능 시점이 소스 코드의 블록 범위와 정확히 같지 않을 수도 있습니다.

.NET GC는 짧게 살아가는 객체가 많다는 세대 가설을 활용합니다. 새 객체는 0세대에서 시작하고, 수집에서 살아남으면 보통 1세대와 2세대로 승격됩니다. 젊은 세대를 자주 수집하고 오래 살아남은 객체가 모인 세대는 상대적으로 덜 검사해 전체 수집 비용을 줄입니다.

  • 0세대

    새로 생성된 단명 객체가 주로 위치하며 가장 자주 수집됩니다.

  • 1세대

    0세대 수집에서 살아남은 객체를 위한 완충 역할을 합니다.

  • 2세대

    오래 살아남은 객체가 위치하며 전체 수집 때 더 큰 비용이 들 수 있습니다.

Unity에서 매 프레임 문자열 결합, 임시 컬렉션, 박싱 같은 관리 할당을 반복하면 수집해야 할 객체가 빠르게 늘어날 수 있습니다. GC가 실행되는 프레임에 작업이 몰리면 프레임 시간이 튀는 GC 스파이크로 체감될 수 있으므로, 프로파일러로 할당 위치와 수집 시간을 확인해야 합니다. 

// 동작 확인용 예시일 뿐, 일반 애플리케이션 코드에서 권장하지 않는다.
GC.Collect();
GC.WaitForPendingFinalizers();

GC.Collect()로 강제 수집할 수 있지만 세대별 수집 정책을 방해하고 불필요한 정지 비용을 만들 수 있어, 측정과 특별한 근거 없이 직접 호출하지 않는 편이 좋습니다.

GC와 Dispose의 차이

GC가 관리하는 주된 대상은 관리 힙의 메모리입니다. 파일 핸들, 소켓, 데이터베이스 연결처럼 운영체제나 외부 시스템의 자원은 사용이 끝난 시점에 결정적으로 반환해야 합니다. 이런 자원을 감싼 객체는 보통 IDisposable을 구현하고, 사용하는 쪽은 using으로 정리 시점을 명확히 합니다. 

using (FileStream stream = new FileStream("data.txt", FileMode.Open))
{
    // 파일 사용
} // 블록을 벗어나며 Dispose() 호출
  • GC: 도달할 수 없는 관리 객체의 메모리를 런타임이 나중에 회수합니다.
  • Dispose: 파일·소켓 같은 자원을 더 이상 쓰지 않는 시점에 즉시 해제합니다.

finalizer는 해제되지 않은 네이티브 자원의 마지막 안전망으로 쓰일 수 있지만 실행 시점과 실행 여부에 의존해서는 안 됩니다. 일반 객체의 소멸 알림이나 핵심 로직을 finalizer에 두지 않고, 소유한 자원은 Dispose로 정리하는 것이 핵심입니다.

언리얼 UObject와 UPROPERTY

C++ 언어 자체에는 추적형 GC가 없지만 언리얼 엔진은 UObject 계열 객체를 위한 별도의 GC를 제공합니다. AActor, UActorComponent, UTexture, UMaterial, UUserWidget 등이 여기에 포함됩니다. 일반 C++ 클래스는 이 관리 대상이 아닙니다.

UObject는 일반 new 대신 객체 종류와 생성 시점에 맞는 엔진 API로 생성합니다.

UMyObject* Object = NewObject<UMyObject>(this);

AEnemy* Enemy = GetWorld()->SpawnActor<AEnemy>(
    EnemyClass,
    SpawnLocation,
    SpawnRotation
);

MeshComponent = CreateDefaultSubobject<UStaticMeshComponent>(
    TEXT("MeshComponent")
);

UObject 참조를 멤버로 오래 유지하려면 엔진이 그 관계를 추적할 수 있게 리플렉션 시스템에 알려야 합니다. UE5 코드에서는 보통 UPROPERTY()와 TObjectPtr를 함께 사용합니다. 

UCLASS()
class MYGAME_API UInventoryComponent : public UActorComponent
{
    GENERATED_BODY()

private:
    UPROPERTY()
    TObjectPtr<UItemData> EquippedItem;
};

UInventoryComponent::UInventoryComponent()
{
    EquippedItem = NewObject<UItemData>(this);
}

UInventoryComponent가 살아 있고 이 프로퍼티가 UItemData를 참조하면 GC가 해당 경로를 인식합니다. 반대로 추적되지 않는 원시 포인터만 보관하면 C++ 변수에는 주소가 남아 있어도 객체의 생존을 보장하지 못합니다. 다만 함수 안의 짧은 로컬 참조처럼 객체가 수집될 기회 없이 즉시 사용하는 경우까지 무조건 UPROPERTY가 필요한 것은 아닙니다. 핵심은 GC가 실행될 수 있는 시간 동안 참조를 보관하는 방식입니다.

Outer와 Actor 수명

NewObject<UMyObject>(this)에서 this는 새 객체의 Outer입니다. Outer는 이름 경로, 포함 관계, 패키징과 수명 관리 맥락에 중요하지만, 일반적인 소유 포인터처럼 Outer 지정만으로 모든 객체의 생존이 자동 보장된다고 단정해서는 안 됩니다. 지속적으로 보관할 객체라면 추적 가능한 UPROPERTY 참조도 명시하는 것이 안전합니다.

월드에 존재하는 Actor는 C++ delete가 아니라 Destroy()로 제거를 요청합니다.

Enemy->Destroy();

if (IsValid(Enemy))
{
    Enemy->Attack();
}

Destroy()는 Actor를 제거 대상으로 표시하고 월드의 적절한 시점에 제거되도록 합니다. 호출 직후 기존 포인터가 자동으로 nullptr이 되는 것은 아닙니다. 따라서 UObject 계열을 다시 사용하기 전에는 단순한 널 검사보다 파괴 대기 상태까지 고려하는 IsValid()를 사용하는 편이 적절합니다.

강한·약한·소프트 참조

참조는 “대상을 살려 둘 것인가”와 “이미 메모리에 있어야 하는가”에 따라 선택할 수 있습니다.

  • TObjectPtr

    UPROPERTY로 추적되는 로드된 UObject의 강한 참조입니다. 소유 객체가 사용하는 필수 대상을 보관할 때 적합합니다.

  • TWeakObjectPtr

    대상의 생존을 보장하지 않는 약한 참조입니다. 대상이 제거될 수 있는 타깃 추적이나 일시적 UI 참조에 유용합니다.

  • TSoftObjectPtr

    객체 경로를 기반으로 하는 소프트 참조입니다. 에셋을 항상 메모리에 올리지 않고 필요할 때 로드하는 흐름에 사용합니다.

UPROPERTY()
TObjectPtr<AEnemy> CurrentEnemy;

UPROPERTY()
TWeakObjectPtr<AEnemy> TargetEnemy;

UPROPERTY(EditAnywhere)
TSoftObjectPtr<UTexture2D> IconTexture;

약한 참조는 사용 전에 IsValid()로 확인해야 합니다. 소프트 참조는 경로를 저장하므로 실제 객체가 필요할 때 동기 또는 비동기 로딩 정책을 별도로 결정해야 합니다. 즉 세 타입은 문법만 다른 포인터가 아니라 수명과 로딩 의도를 표현합니다.

일반 C++ 객체는 GC 대상이 아니다

UObject를 상속하지 않은 일반 C++ 타입은 언리얼 GC가 관리하지 않습니다. 직접 new했다면 직접 해제해야 하지만, 가능하면 값 타입이나 RAII 스마트 포인터로 소유권을 코드에 드러내는 편이 안전합니다.

class FInventoryData
{
public:
    int32 Gold = 0;
};

// 값으로 보관: 가장 단순한 선택
FInventoryData Data;

// 단독 소유
TUniquePtr<FInventoryData> UniqueData =
    MakeUnique<FInventoryData>();

// 실제로 공동 소유가 필요할 때
TSharedPtr<FInventoryData> SharedData =
    MakeShared<FInventoryData>();

TUniquePtr와 TSharedPtr는 참조 추적 GC가 아니라 스코프와 소유권 규칙에 따라 객체를 정리하는 스마트 포인터입니다. UObject에는 UObject용 참조 타입을, 일반 C++ 객체에는 값과 RAII를 사용한다는 경계를 구분해야 합니다.

정리

  • 추적형 GC는 GC Root에서 도달할 수 없는 객체를 수집하므로 순환 참조도 처리할 수 있습니다.
  • null 대입은 참조를 끊을 뿐 즉시 메모리를 해제하지 않으며, 살아 있는 불필요한 참조도 관리해야 합니다.
  • C#의 세대별 GC는 단명 객체를 자주 수집해 비용을 줄이지만 반복 할당은 GC 스파이크를 만들 수 있습니다.
  • GC와 IDisposable은 역할이 다르며 외부 자원은 Dispose로 정리 시점을 보장해야 합니다.
  • 언리얼 GC는 UObject 계열을 관리하며, 오래 보관하는 참조는 UPROPERTY와 적절한 UObject 참조 타입으로 표현합니다.
  • Outer는 포함 관계에 중요하지만 모든 생존을 대신 보장하는 강한 참조로 생각하면 안 됩니다.
  • Actor는 Destroy()로 제거하고, 이후 접근 전에는 IsValid()로 상태를 확인합니다.
  • 일반 C++ 객체는 GC 대상이 아니므로 값 타입이나 RAII 스마트 포인터로 관리합니다.

결국 안전한 메모리 관리의 핵심은 “자동으로 지워지는가”보다 “누가 이 객체를 소유하고, 무엇이 생존을 보장하며, 언제 자원을 반환해야 하는가”를 명확히 하는 것입니다. C#과 언리얼은 서로 다른 규칙을 제공하지만, 참조와 소유권의 의도를 코드에 드러내야 한다는 점은 같습니다.
