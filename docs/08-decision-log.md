# 결정 로그

프로젝트의 구조나 운영 방식에 영향을 주는 결정을 시간순으로 기록한다. 각 항목에는 배경, 결정, 이유, 다시 검토할 조건을 남긴다.

## 2026-07-07 — 빌드 없는 정적 사이트로 시작

- 배경: 저장소가 빈 `index.html` 중심의 초기 상태이며, 우선 반복 개발을 위한 기준이 필요하다.
- 결정: 루트의 `index.html`과 `styles.css`만으로 첫 화면을 구성하고 문서 기반 하네스를 마련한다.
- 이유: GitHub Pages에 바로 배포할 수 있고 의존성, 설정, 유지 비용이 가장 작다.
- 재검토 조건: 상세 페이지와 글이 늘어 공통 레이아웃 복사, 목록 관리, Markdown 변환이 반복될 때.

## 2026-07-07 — 콘텐츠와 가독성을 시각 효과보다 우선

- 배경: 포트폴리오는 작업의 맥락과 기여를 빠르게 전달해야 한다.
- 결정: 시스템 글꼴, 제한된 색상, 단순한 카드와 반응형 그리드를 사용하고 애니메이션은 추가하지 않는다.
- 이유: 첫 로드, 접근성, 유지보수성과 핵심 정보의 가시성을 높인다.
- 재검토 조건: 실제 콘텐츠가 갖춰진 뒤 사용성 검증에서 명확한 필요가 확인될 때.

## 2026-07-07 — Astro/Vite는 향후 선택지로 유지

- 배경: 장기적으로 Markdown 글과 반복 레이아웃이 필요할 수 있다.
- 결정: 현재는 도입하지 않는다. 콘텐츠 중심 정적 생성이 필요하면 Astro를, JavaScript 번들링 중심 요구가 생기면 Vite를 검토한다.
- 이유: 필요가 생기기 전에 도구 복잡성을 추가하지 않기 위해서다.
- 재검토 조건: `docs/04-tech-architecture.md`의 전환 기준을 충족할 때.

## 2026-07-08 — 사용자 선택형 다크·화이트 테마 제공

- 배경: 기존 다크 화면을 유지하면서 밝은 환경에서도 읽기 편한 화면 선택지가 필요하다.
- 결정: 다크 테마를 기본값으로 유지하고, 베이지 배경과 검정 강조색의 화이트 테마를 헤더 버튼으로 전환한다. 선택은 `localStorage`에 저장한다.
- 이유: 별도 의존성이나 빌드 단계 없이 사용자가 원하는 대비를 계속 사용할 수 있다.
- 재검토 조건: 운영체제 테마 자동 추종이나 여러 색상 테마에 대한 요구가 확인될 때.

## 2026-07-08 — TIL 글은 하위 페이지로 제공

- 배경: 학습 글을 공유 가능한 URL로 열고, 본문 분량이 늘어나도 홈 화면을 가볍게 유지할 필요가 있다.
- 결정: TIL / 블로그 섹션의 카드는 `/til/{post-slug}.html` 하위 페이지로 연결한다.
- 이유: 프레임워크와 라우팅 없이도 글마다 독립된 문서 구조와 URL을 제공할 수 있다.
- 재검토 조건: 글이 늘어나 공통 헤더와 글 목록 관리가 반복될 때.

## 2026-07-08 — TIL 전체 목록 페이지 추가

- 배경: 앞으로 TIL과 공부 기록을 계속 추가하면 홈 화면에 모든 글을 노출하기 어렵고, 글 전체를 탐색할 별도 진입점이 필요하다.
- 결정: `/til/index.html`에 전체 글 목록 페이지를 두고, 홈의 TIL / 블로그 섹션은 최근 글 일부만 보여준 뒤 `모든 글 보기` 링크로 목록 페이지에 연결한다.
- 이유: 정적 HTML/CSS 구조를 유지하면서도 홈 화면을 가볍게 유지하고, 글이 늘어날 때 목록을 한곳에서 관리할 수 있다.
- 재검토 조건: 글 목록 수동 관리가 반복되거나 Markdown 기반 자동 목록 생성이 필요해질 때.

## 2026-07-08 — 프로젝트 상세 사례는 하위 페이지로 제공

- 배경: Roguelike Project README에는 설계 목표, 구현 구조, 개발 기록, 트러블슈팅이 포함되어 홈 카드나 팝업에 담기에는 분량이 길다.
- 결정: 홈의 Projects 섹션에는 요약 카드를 두고, 상세 내용은 `/projects/roguelike-project.html` 하위 페이지로 연결한다.
- 이유: 공유 가능한 URL과 목차가 있는 문서 구조를 제공하면서 홈 화면은 가볍게 유지할 수 있다.
- 재검토 조건: 프로젝트 상세 페이지가 늘어나 공통 레이아웃 복사와 목록 관리가 반복될 때.

## 2026-07-10 — TIL 댓글은 giscus와 GitHub Discussions로 제공

- 배경: 정적 GitHub Pages 구조를 유지하면서 TIL 글마다 독립적인 댓글 공간이 필요하다.
- 결정: 각 TIL 글 하단에 giscus 임베드 스크립트를 추가하고, `pathname` 매핑으로 글 URL별 GitHub Discussions 댓글을 연결한다.
- 이유: 별도 서버나 데이터베이스 없이 GitHub 계정 기반 댓글, 반응, 관리를 제공할 수 있고 현재 저장소 운영 흐름과 잘 맞는다.
- 재검토 조건: GitHub 로그인이 없는 방문자의 댓글 참여가 중요해지거나, 외부 스크립트 의존을 제거해야 할 때.

## 2026-07-10 — 코드블록 구문 강조는 로컬 스크립트로 처리

- 배경: TIL 글의 C++ 코드블록이 늘어나면서 키워드, 타입, 문자열, 숫자, 주석을 시각적으로 구분할 필요가 있다.
- 결정: 외부 패키지나 빌드 도구 없이 `code-highlight.js`와 `styles.css` 토큰 색상만으로 코드블록을 강조한다.
- 이유: 정적 사이트 구조와 의존성 없는 운영 방식을 유지하면서 코드 가독성을 높일 수 있다.
- 재검토 조건: 지원해야 할 언어가 크게 늘어나거나 정확한 구문 분석이 필요해져 Prism.js 같은 검증된 하이라이터가 더 적합해질 때.

## 2026-07-10 — 알고리즘 풀이 노트는 사고 과정까지 확장해 기록

- 배경: 정답 코드만으로는 어떤 접근에서 막혔고 DP 상태와 순회 방향을 어떻게 결정했는지 전달하기 어렵다.
- 결정: 로컬 풀이 노트를 TIL로 옮길 때 문제 이해, 초기 접근, 전환점, 점화식, 예시, 코드, 복잡도와 배운 점을 갖춘 독립 글로 확장한다.
- 이유: 결과뿐 아니라 문제 해결 과정과 기술적 판단을 포트폴리오에서 명확하게 보여줄 수 있다.
- 재검토 조건: 알고리즘 글의 수가 늘어 공통 글 구조를 템플릿이나 콘텐츠 도구로 관리해야 할 때.
